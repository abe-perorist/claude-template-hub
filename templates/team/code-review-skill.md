# スキル: コードレビュー

PRまたは指定されたコードに対して、チェックリストに基づいたコードレビューを実行する。

---

## ワークフロー

### Step 1: 対象を特定する

- 引数にPR番号が渡された場合: `gh pr view {番号} --json files,diff,title,body` で内容を取得する
- 引数なしの場合: `git diff main...HEAD` で現在のブランチの変更を取得する

### Step 2: 変更内容を把握する

- 変更されたファイル一覧を確認する
- 変更の目的（PRタイトル・説明）を確認する
- 変更量が大きい場合はファイルごとに分割してレビューする

### Step 3: チェックリストに従ってレビューする

#### 正確性
- [ ] ロジックに明らかなバグはないか
- [ ] エッジケース（null・空配列・境界値）が考慮されているか
- [ ] エラーハンドリングが適切か

#### セキュリティ
- [ ] ユーザー入力のバリデーションがあるか
- [ ] SQLインジェクション・XSS・コマンドインジェクションの余地はないか
- [ ] シークレット・APIキーがコード中にハードコードされていないか

#### 可読性・保守性
- [ ] 変数名・関数名が意図を正確に表しているか
- [ ] 複雑なロジックにコメントがあるか
- [ ] 関数が単一責任になっているか（肥大化していないか）

#### テスト
- [ ] 変更に対応するテストが追加・更新されているか
- [ ] テストが実際に変更をカバーしているか

#### チーム規約
- [ ] ブランチ名・コミットメッセージがルールに従っているか
- [ ] 不要な `console.log` / デバッグコードが残っていないか
- [ ] `.env` ファイルなど機密情報がコミットされていないか

### Step 4: フィードバックを出力する

以下の形式でまとめる：

```markdown
## コードレビュー結果

### 概要
{変更の目的と全体的な印象を1〜3行で}

### 問題点
#### 🔴 必須修正
- {ファイル名}:{行番号} — {問題の説明と修正案}

#### 🟡 推奨修正
- {ファイル名}:{行番号} — {改善提案}

#### 💬 コメント・質問
- {疑問点や議論したいこと}

### 良かった点
- {良い実装・工夫へのコメント}
```

---

## 使用例

```
/review              # 現在のブランチの変更をレビュー
/review 123          # PR #123 をレビュー
```
