# スキル: TDDワークフロー (Red-Green-Refactor)

テスト駆動開発のRed-Green-Refactorサイクルに従って機能を実装する。
テストを先に書き、テストが失敗することを確認してから実装コードを書く。

---

## ワークフロー

### Step 1: 要件を整理する

- ユーザーの要求を確認し、実装すべき機能の仕様を明確にする
- 機能を小さなテスト可能な単位に分解する
- 実装順序を決定する（依存関係の少ないものから）

### Step 2: RED - テストを書く（失敗するテスト）

- 最初のテストケースを書く
- テストを実行し、**失敗することを確認する**
- テストが失敗しない場合は、テストが正しく書かれているか見直す

```
テスト実行コマンド例:
- JavaScript/TypeScript: npx jest / npx vitest
- Python: pytest
- Go: go test ./...
- Rust: cargo test
```

> 重要: テストが RED（失敗）であることを確認するまで次のステップに進まない。

### Step 3: GREEN - 最小限の実装を書く

- テストを通すための**最小限のコード**を書く
- 過剰な設計やエッジケース対応はこの段階では行わない
- テストを実行し、**成功することを確認する**

> 重要: テストが GREEN（成功）であることを確認するまで次のステップに進まない。

### Step 4: REFACTOR - コードを改善する

- テストが通った状態を維持しつつ、コードを整理する
- 重複の除去、命名の改善、構造の整理を行う
- リファクタリング後にテストを再実行し、**すべて成功することを確認する**
- リファクタリングが不要であればスキップしてよい

### Step 5: サイクルを繰り返す

- Step 1で分解した次のテストケースに進む
- Step 2〜4を繰り返す
- すべての要件がテストでカバーされるまで続ける

### Step 6: 最終確認

- すべてのテストを実行し、全テストが成功することを確認する
- テストカバレッジを確認する（ツールがある場合）
- 実装結果をサマリーとして出力する

```markdown
## TDD実装結果

### 実装した機能
- {機能の説明}

### テスト一覧
| テスト名 | 状態 |
|---------|------|
| {テスト名} | PASS |

### サイクル数
- RED-GREEN-REFACTOR: {N}回

### 備考
- {特記事項があれば}
```

---

## ルール

- **テストファーストを厳守する**: 実装コードを先に書かない
- **各フェーズでテストを実行する**: RED→実行→GREEN→実行→REFACTOR→実行
- **最小限の実装**: GREEN フェーズでは、テストを通す最小限のコードだけを書く
- **1テストずつ**: 一度に複数のテストを追加しない。1つずつサイクルを回す
- **既存テストを壊さない**: 新しいテストを追加しても、既存のテストはすべて通ること

---

## 使用例

```
/tdd ユーザー登録機能を実装して
/tdd calculateTotal関数にディスカウント計算を追加して
/tdd APIのバリデーションロジックをテスト付きで書いて
```
